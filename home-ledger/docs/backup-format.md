# Current Backup Format

This document describes the existing full-backup format and restore behavior for the local app. It is not a cloud import schema.

## Sources Inspected

- `backend/domain/backup.js`: backup envelope creation, validation, file payload normalization, blocked file rules, checksum helpers, restore reconciliation.
- `backend/domain/model.js`: app id, backup version, product/version constants, data sanitization, relationship validation.
- `frontend/app.js`: backup creation, JSON download, restore confirmation, file restoration, checksum checks, tutorial restore handling.
- `backend/storage/document-storage.js`: file record persistence and restored file records.
- `desktop/main.cjs`: desktop backup save limits and attachment storage limits.
- `tests/backup.test.mjs`: accepted and rejected backup behavior.
- `tests/tutorial-data.test.mjs`: tutorial backup behavior.
- `tests/ui-copy.test.mjs`: backup and restore copy expectations.

## Confirmed Envelope Fields

Source: `createBackupEnvelope(data, files, missingFiles, createdAt)` in `backend/domain/backup.js`.

```js
{
  app: "home-basis-tracker",
  productName: "Home Basis Tracker",
  productVersion: "0.1.0",
  exportType: "full-backup",
  backupVersion: 1,
  createdAt: "...",
  data: { ...sanitized app data... },
  files: [ ...file payloads... ],
  missingFiles: [ ...missing file records... ],
}
```

Confirmed constants:

- `BACKUP_APP_ID = "home-basis-tracker"`
- `BACKUP_VERSION = 1`
- `EXPORT_PRODUCT_NAME = "Home Basis Tracker"`
- `EXPORT_PRODUCT_VERSION = "0.1.0"`
- `MAX_BACKUP_FILE_SIZE = 500 * 1024 * 1024`
- `MAX_DOCUMENT_FILE_SIZE = 25 * 1024 * 1024`

Backups are JSON files. `frontend/app.js` writes the serialized envelope plus a trailing newline.

## Data Payload

The `data` payload is sanitized with `sanitizeData(data)` before export and again during validation. It uses the current local app root:

- `vendors`
- `properties`
- `projects`
- `expenses`
- `documents`
- `followUpOverrides`

The backup payload preserves the local data shape. It does not embed a future SaaS model, cloud workspace model, or API representation.

## Files Payload

Source: `normalizeBackupFileRecords(files, documents)` in `backend/domain/backup.js` and `buildFullBackup()` in `frontend/app.js`.

Each file payload can include:

| Field | Rule |
| --- | --- |
| `documentId` | Clean string. Must identify a document with `hasFile: true`. |
| `fileId` | Clean string. May match the document file id. File ids must be unique. |
| `fileName` | Safe filename from `getSafeDownloadFileName`. Defaults to `Attached file`. |
| `mimeType` | Clean MIME type. Defaults to `application/octet-stream`. |
| `fileSize` | Nonnegative number. |
| `fileLastModified` | Optional source file metadata. |
| `fileStoredAt` | Optional stored timestamp. |
| `sha256` | Optional lowercase 64-character hex hash. Invalid hashes normalize to blank. |
| `dataUrl` | Optional string data URL containing file contents. |

Backup validation rejects:

- Duplicate file payloads for the same `documentId`.
- Duplicate `fileId` values.
- File payloads that do not correspond to a document with `hasFile: true`.
- File payloads with neither `documentId` nor `fileId`.

During backup creation, `frontend/app.js` reads each stored document file, converts it to a data URL, and attempts to compute SHA-256 with Web Crypto. If Web Crypto is unavailable during backup creation, the hash can be blank.

## Missing Files Payload

Source: `buildFullBackup()` in `frontend/app.js`.

`missingFiles` records document file metadata that could not be included in the backup. Confirmed fields include:

- `documentId`
- `fileId`
- `fileName`
- `reason`

Reasons are generated by the UI flow, for example missing stored file content or sample/tutorial metadata. `summarizeBackupEnvelope(backup)` reports:

- `missingFilesCount`
- `expectedFilesMissingFromBackup`

## Validation And Versioning

Source: `validateBackupEnvelope(backup)` in `backend/domain/backup.js`.

Confirmed restore validation rejects:

- Missing or wrong `app`.
- Missing `data`.
- Backup versions newer than the app supports.
- Duplicate ids for vendors, properties, projects, expenses, or documents.
- Broken project-to-property relationships.
- Broken expense-to-property relationships.
- Expense project links where the project belongs to another property.
- Broken document-to-expense relationships.
- Document property/project links that conflict with the linked expense.
- Document project links where the project belongs to another property.
- Invalid file payload relationships or duplicates.

Validation returns sanitized data and normalized files. It does not create cloud ids or alter the backup version.

## Restore File Handling

Source: `prepareBackupRestore(backup, options)` in `frontend/app.js`.

Confirmed behavior:

- Backup files larger than `500 MB` are rejected before parse.
- Invalid JSON is rejected.
- Restore shows a confirmation with record counts and file coverage.
- Existing app records are replaced by the restored dataset after confirmation.
- Existing stored file ids not imported by restore are deleted best-effort after successful restore.
- Restore writes new stored file ids for imported files. It does not reuse the backup file id as the new storage id.
- If file restoration fails for one document, the document can be kept with file metadata stripped and `fileStatusNote` explaining why.
- If restore fails after saving new files, newly saved files are deleted best-effort.

For each document with `hasFile: true`, restore tries to find a matching backup file payload. If file content cannot be restored, `stripDocumentFileMetadata(document, reason)` clears:

- `hasFile`
- `fileId`
- `mimeType`
- `fileSize`
- `fileLastModified`
- `fileStoredAt`

It keeps or annotates filename/status information with a reason.

## Data URL And Checksum Rules

Source: `backupDataUrlIsWithinLimit(dataUrl)`, `backupFileChecksumMatches(blob, expectedHash)`, and `dataUrlToBlob(dataUrl)`.

Confirmed behavior:

- Empty or missing data URLs are allowed as metadata but do not restore file content.
- Data URLs must start with `data:` to decode.
- Data URLs are rejected when their string length exceeds `MAX_BACKUP_DATA_URL_LENGTH`, which is based on the 25 MB document limit plus encoding overhead.
- Decoded blobs over `25 MB` are rejected for document restore.
- If `sha256` is blank, checksum validation passes.
- If `sha256` is present but Web Crypto is unavailable, checksum validation fails.
- If `sha256` is present and does not match the decoded blob, file restore is skipped for that document.

## Blocked File Rules

Source: `BLOCKED_BACKUP_FILE_EXTENSIONS`, `BLOCKED_BACKUP_MIME_PREFIXES`, and `isBlockedBackupFile(record)` in `backend/domain/backup.js`.

Blocked extensions:

- `.app`
- `.bat`
- `.cmd`
- `.command`
- `.exe`
- `.js`
- `.jse`
- `.msi`
- `.ps1`
- `.scr`
- `.sh`
- `.vbs`
- `.wsf`

Blocked MIME prefixes:

- `application/x-ms`
- `application/x-sh`
- `text/javascript`

Blocked file payloads are not restored as stored files. The document record can remain with file metadata stripped and a status note.

## Tutorial Backup Behavior

Source: `buildFullBackup()` and `prepareBackupRestore()` tutorial branches in `frontend/app.js`; `tests/tutorial-data.test.mjs`.

Confirmed behavior:

- Tutorial backups validate without real file payloads.
- Tutorial document file metadata is sample-only.
- Tutorial restore strips attached file metadata with a note that content was not restored inside the tutorial workspace.
- Tutorial mode should not be treated as proof that real document bytes exist.

## Export And Restore Copy Constraints

Source: `tests/ui-copy.test.mjs`.

Confirmed behavior:

- Export and backup copy distinguishes a review packet from a full backup.
- Restore copy names the workspace being replaced, shows counts, and identifies attached file coverage.
- Public copy avoids prohibited product claims and avoids implying the app makes professional conclusions.

## Inferred Or Ambiguous Behavior

- `missingFiles` is informational. The restore path does not use it as a source of file content.
- Backup validation sanitizes the data payload, but restore remains a replace operation rather than a merge/import operation.
- Duplicate legacy ids reject restore today. A future migration importer will need a different duplicate-handling policy if users import multiple backups into one workspace.
- Empty checksum values are accepted to support backups created without Web Crypto. This preserves compatibility but creates a weaker integrity signal for those files.
- File ids are local storage ids, not durable cloud object ids.
