# Current Follow-up Rules

This document captures how the local app generates, displays, and suppresses follow-up items. It is descriptive and should be used as migration source behavior.

## Sources Inspected

- `backend/domain/model.js`: `getRecordFollowUps`, `getRecordFollowUpsForSurface`, project completeness, property/project/expense/document follow-up generators, override filtering.
- `frontend/app.js`: dashboard Needs attention subtab, project open-item expansion, follow-up resolution modals, override save flow, project completeness note UI.
- `tests/model.test.mjs`: follow-up generation, project-centered document follow-ups, missing-document wording, expense support wording, override behavior, completeness overrides.
- `tests/ui-copy.test.mjs`: dashboard subtab and dialog accessibility/copy expectations.

## Confirmed Generation Pipeline

Source: `getRecordFollowUps(data, options = {})` in `backend/domain/model.js`.

Confirmed behavior:

- Input data is sanitized before follow-ups are generated.
- Follow-ups are generated from:
  - properties
  - projects
  - expenses
  - documents
- Follow-up ids are de-duplicated.
- Items whose ids appear in `data.followUpOverrides` are removed.
- Default severity is `medium`.
- Default priority is `50`.
- Default surfaces are `dashboard` and `export`.
- Items are sorted by priority and then id.

`getRecordFollowUpsForSurface(data, surface)` filters generated items by `item.surfaces`.

## Property Follow-ups

Source: property section inside `getRecordFollowUps(data, options)`.

Confirmed rules:

| Type | Id pattern | Trigger | Label |
| --- | --- | --- | --- |
| `property-missing-purchase-date` | `property:{propertyId}:purchase-date` | Property has no `purchaseDate`. | `Add purchase date` |
| `property-missing-purchase-price` | `property:{propertyId}:purchase-price` | Property has no positive `purchasePrice`. | `Add purchase price` |

Property follow-ups point to the property editor and include property name context.

## Project Completeness

Source: `getProjectCompleteness(data, project)` and `buildProjectCompleteness(...)`.

Confirmed checks:

- Linked to a property.
- Project description or notes added.
- Start date added.
- Completion date handled for completed or archived projects.
- Contractor or vendor identified.
- Costs linked.
- Supporting documents linked.
- Receipt/invoice files linked.
- Cost type review choices made.
- Expected document types covered.

The completeness score is derived from completed checks over total checks. `completenessOverrideNote` overrides project completeness to 100 percent, suppresses project follow-ups, and reports the project as marked complete with note.

## Project Follow-ups

Source: `getProjectFollowUpItems(records, project)` in `backend/domain/model.js`.

Project follow-ups are suppressed when `project.completenessOverrideNote` is present.

Confirmed project item types:

| Type | Trigger | Label behavior |
| --- | --- | --- |
| `project-missing-vendor` | No project vendor/contractor and no linked expense vendor. | `Add project vendor` |
| `project-missing-dates` | Missing start date, or completed/archived project lacks completion date. | `Add project dates`, `Add start date`, or `Add completion date` |
| `project-missing-scope` | No `scopeSummary` and no project notes. | `Add project description` |
| `project-missing-supporting-documents` | Missing expected project document type. | Specific labels such as `Add contract or estimate`, `Add before/after photo`, or `Add permit or approval` |

Expected project document types come from `getExpectedProjectDocumentTypes(project, expenses, totalSpend)`:

- Expenses exist: receipt or invoice evidence is expected.
- Permit number or permit-likely category: permit or approval may be expected.
- Vendor/contractor or total spend of at least `1000`: contract or estimate may be expected.
- Completed or archived project: before/after photo may be expected.
- Expenses exist: payment proof may be expected.

Project follow-up generation intentionally surfaces specific missing project document types, not a single bundled sentence. Tests confirm separate follow-ups for contract/estimate and before/after photo.

Receipt/invoice and payment record expectations are used for completeness, but project missing-document follow-up generation filters those expected types out so expense/document evidence flows can create more specific items.

## Expense Follow-ups

Source: `getExpenseFollowUpItems(records, expense)` in `backend/domain/model.js`.

Confirmed expense item types:

| Type | Trigger | Label behavior |
| --- | --- | --- |
| `expense-missing-vendor` | Expense has no `vendorId` and no vendor text. | `Add vendor` |
| `expense-review-treatment` | `classification` is `unclear / ask CPA`. | `Review cost type` |
| `expense-missing-document-support` | Expense status is `no document yet` or `needs follow-up`, and no linked receipt/invoice support exists. | `Add receipt for {vendor/support label}` or invoice equivalent. |
| `expense-documented-without-support` | Expense says receipt/invoice attached but no linked uploaded support exists. | `Upload receipt for {vendor/support label}` or invoice equivalent. |

Expected support type is determined by `getExpectedExpenseDocumentType(expense)`: invoice is expected when `documentationStatus` is `invoice attached`; otherwise receipt is expected.

If a linked receipt/invoice document record exists but is missing its file, expense-level upload follow-up is suppressed and document-level missing-file follow-up is used instead. `tests/model.test.mjs` confirms this behavior.

## Document Follow-ups

Source: `getDocumentFollowUpItem(records, documentRecord, options)` in `backend/domain/model.js`.

Confirmed document item types:

| Type | Trigger | Label behavior |
| --- | --- | --- |
| `document-missing-attached-file` | Document record exists without stored file. | `Upload {document type} file` |
| `document-restored-without-file-content` | Document file content was restored/skipped/missing/not included. | `Restore or upload {document type} file` |
| `document-tutorial-metadata-only` | Tutorial metadata-only document file. | `Upload {document type} file` |

Documents with `hasFile: true` do not generate document follow-ups.

Document-related follow-ups are no longer surfaced through the Documents tab. `tests/model.test.mjs` confirms `getRecordFollowUpsForSurface(data, "documents")` returns no document-related items, while the Projects surface includes document missing-file items.

## Project-centered Resolution Behavior

Source: `frontend/app.js`, especially project row expansion, dashboard Needs attention, and follow-up modal handlers.

Confirmed behavior:

- The Dashboard has `Recent activity` and `Needs attention` subtabs.
- Needs attention content is rendered as a compact table/grid.
- Project records show open item counts.
- Clicking project open-item counts expands downward within the project row/record.
- Clicking an item opens a focused resolution modal rather than jumping unexpectedly to another tab.
- Dashboard `Open` actions use modal/detail behavior while keeping the Dashboard underneath.
- Follow-up item modals include record context and the action required to resolve the issue.

## Override Behavior

Source: `followUpOverrides` model, override save flow in `frontend/app.js`, and `tests/model.test.mjs`.

Confirmed behavior:

- A user can override a follow-up item and mark it complete.
- The override can include a note.
- The override is persisted in `followUpOverrides`.
- A persisted override removes only the follow-up with the matching id.
- Overrides do not mutate the underlying source record, expense, document, or project field.
- Project `completenessOverrideNote` is separate from item-level `followUpOverrides` and marks the whole project completeness checklist complete by note.

## Recompute Expectations After Migration

Confirmed source behavior:

- Follow-ups are derived from current records plus overrides.
- Open item counts should be recomputed, not imported as static counts.
- `followUpOverrides` should be imported before recomputing so overridden ids remain suppressed.
- Project completeness should be recomputed from project, expense, document, and override state.

## Inferred Or Ambiguous Behavior

- Follow-up ids are deterministic strings based on local ids and rule names. If migrated records receive new primary ids, override matching will break unless legacy ids are preserved for follow-up id reconstruction.
- `classification` still uses legacy value `unclear / ask CPA`, even though UI copy now uses `Not sure, review later`. Migration must decide whether to preserve legacy enum values or introduce a mapped enum.
- `completenessOverrideNote` suppresses all project follow-ups, including expense/document problems under that project. That is confirmed behavior but may deserve product review before cloud migration.
- Document follow-ups are project-centered in the current UI, but documents still carry file metadata and can be opened/edited from document flows.
